Objective-C之 atomic 和 noatomic 原理分析

Feb 27, 2019

前言

开发中使用@property声明属性时，经常用到atomic与nonatomic两个关键字，最初接触时只是知道atomic是线程安全的，nonatomic是线程不安全的，但两者究竟有何区别?

区别

atomic
  • 是默认的
  • 是原子性的，系统自动生成的 setter 和 getter 方法会进行加锁操作
  • 需要消耗大量的资源，使系统变慢
nonatomic
  • 不是默认的
  • 是非原子性的，系统自动生成的 setter 和 getter 方法不会进行加锁操作
  • 线程不安全，但更快，适合内存小的移动设备
  • 如果两个线程访问同一个属性，会出现无法预料的结果

atomic

atomic是Objc使用的一种线程保护技术，多线程下将属性设置为atomic可以保证读取数据的一致性，因为它将保证数据只被一个线程占用，也就是说一个线程在对数据进行读写操作时，会使用自旋锁锁住该属性，不允许其他线程再进行读写操作了。而这种机制是耗费系统资源的，所以在iPhone这种小型设备上，如果过没有使用多线程间的通讯，那么nonatomic是一个非常好的选择。

总结

很多文章都谈到atomic和nonatomic的区别时，都说atomic是线程安全的，其实这个说法并不准确。

atomic只是对 setter 和 getter 进行了加锁操作，这种安全只是对 setter 和 getter 的读写安全，并非真正意义上的线程安全。
线程安全除了读写操作之外还有其他操作。(比如当一个线程对一个array进行removeObject操作时，又有另一个线程同时对array进行for循环操作时，可能会直接crash)。所以多线程安全只能靠程序员自己保障，atomic属性和使用多线程安全并没有什么直接的联系。另外，atomic由于加锁也会带来一些性能上的损耗，所以我们在编写iOS代码的时候，一般声明 property 为nonatomic，在需要做多线程安全的场景，需要自己去额外加锁做同步。
